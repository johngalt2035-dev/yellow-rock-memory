Quick Reference: 3 Steps (All Methods)

Step 1

Download & Build

# Mac (with Homebrew + Rust)
git clone https://github.com/
  johngalt2035-dev/yellow-rock-memory.git
cd yellow-rock-memory
cargo build --release
cp target/release/yellow-rock-memory
  /usr/local/bin/

Need Rust? Run: curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh

Step 2

Start the Memory System

yellow-rock-memory \
  --db ~/yellow-rock.db \
  serve --port 9077

That's it. It's running. The API is live at http://localhost:9077

Verify: curl localhost:9077/api/v1/health

Step 3

Train with Your Data

# Create a file: background.json
[
  {"title": "Schedule",
   "content": "Conference call at 3pm"},
  {"title": "Contact info",
   "content": "Their work: 9-5"}
]
# Then import it:
yellow-rock-memory \
  --db ~/yellow-rock.db \
  train background.json

Also works with Markdown files (.md). Done!

Step 1 installed the yellow-rock-memory program on your computer.

Step 2 started the memory system. It's now listening for your AI assistant to store and retrieve messages.

Step 3 loaded background knowledge — schedules, contacts, anything the AI should know about your principal's business context.

Now connect it to your AI assistant (ChatGPT, Claude, Grok, Gemini, or Ollama) and your messaging platform (Signal, Telegram, WhatsApp, or SMS). See the full setup guide for details.

Symbiotic System

Yellow Rock Memory is one half of a symbiotic system. It provides the data backbone while Yellow Rock Protocol provides the communication rules. Both are built upon OpenClaw.

Yellow Rock Memory

Forensic archive. Escalation scoring. Training facility. SHA-256 chain. Per-contact routing. SQLite + FTS5.

Yellow Rock Protocol

BIFF templates. Anti-JADE/DARVO. Cluster B patterns. Legal framework. Escalation management. Emergency triage.

OpenClaw

Multi-agent orchestration. Channel routing (Signal, Telegram, WhatsApp, SMS). Cron scheduling. LLM provider management.

Data flow: Contact → OpenClaw (channel) → AI Agent (protocol) → Yellow Rock Memory (archive + recall) → AI Agent (BIFF response) → OpenClaw (delivery) → Contact

Security Architecture

Yellow Rock Memory enforces a 6-layer defense model to prevent unauthorized access, contact impersonation, and data exfiltration. Every layer is independent — compromising one does not bypass the others.

Environment Variables

VariablePurposeRequired
GRM_API_KEYBearer token for HTTP API authentication. All draft endpoints require this token in the Authorization header.Yes (HTTP mode)
GRM_PRINCIPAL_IDIdentity of the system owner (principal). Used by MCP tools to enforce that only the principal can approve, reject, or send drafts.Yes (MCP mode)

6 Defense Layers

Layer 1: HTTP Bearer Token Auth

All draft management endpoints require a valid Authorization: Bearer <GRM_API_KEY> header. Requests without a valid token receive a 401 response with no information leakage.

Layer 2: MCP Principal ID Enforcement

MCP tool calls enforce GRM_PRINCIPAL_ID — only the declared principal can invoke approve, reject, and send operations. AI agents cannot self-authorize.

Layer 3: DB-Level Contact Block

The database layer rejects any approve or reject operation where reviewer == contact_id (case-insensitive). Prevents the high-conflict counterparty from approving their own drafts even if outer layers are bypassed.

Layer 4: Sanitized Error Messages

All error responses use generic messages ("operation failed", "operation denied"). No internal state, database schema, or stack traces are ever exposed to callers.

Layer 5: Localhost-Only Binding

The HTTP server binds to 127.0.0.1 by default. The API is not network-accessible unless explicitly configured otherwise. MCP runs over stdio with no network exposure.

Layer 6: AI Tool Descriptions

MCP tool descriptions include PRINCIPAL-ONLY directives that instruct the AI agent to gate sensitive operations behind human confirmation. Defense in depth at the prompt layer.

OpenClaw First-Time Setup

On first launch, run the onboarding wizard before starting the gateway:

# 1. Run interactive onboarding
docker compose -f docker-compose.full.yml \
  run --rm --no-deps --entrypoint node \
  openclaw-gateway dist/index.js onboard \
  --mode local --no-install-daemon

# 2. Set gateway config
docker compose -f docker-compose.full.yml \
  run --rm --no-deps --entrypoint node \
  openclaw-gateway dist/index.js config set gateway.mode local
docker compose -f docker-compose.full.yml \
  run --rm --no-deps --entrypoint node \
  openclaw-gateway dist/index.js config set gateway.bind lan

# 3. Start the full stack
docker compose -f docker-compose.full.yml up -d

The onboarding wizard prompts for AI provider keys, generates a gateway token, and creates openclaw.json. Config is bind-mounted from the host so it persists across container rebuilds.

Connect Memory to OpenClaw

Inside the Docker network, the memory daemon is reachable at http://memory:9077. Point OpenClaw's memory backend there:

# Via CLI
docker compose -f docker-compose.full.yml \
  run --rm --profile cli openclaw-cli \
  config set agents.defaults.memorySearch.remote.baseUrl \
  "http://memory:9077/v1"

Or add directly to openclaw.json:

{
  "agents": {
    "defaults": {
      "memorySearch": {
        "remote": { "baseUrl": "http://memory:9077/v1" }
      }
    }
  }
}

Security

Non-Root Containers

Memory runs as yrm. OpenClaw runs as node (uid 1000). CLI drops NET_RAW/NET_ADMIN capabilities.

Localhost Only

All ports bound to 127.0.0.1 — not network-accessible. Use a reverse proxy (Caddy, nginx) for production exposure with TLS.

Bearer Token Auth

Draft endpoints require GRM_API_KEY. Gateway requires OPENCLAW_GATEWAY_TOKEN. No key = no access (fail-closed).

Production: Reverse Proxy

For network exposure, add TLS termination via Caddy, nginx, or Traefik:

# Caddyfile
memory.yourdomain.com {
    reverse_proxy memory:9077
}
gateway.yourdomain.com {
    reverse_proxy openclaw-gateway:18789
}
